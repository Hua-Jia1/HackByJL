# 02 · 业务流程(流量怎么走的)

> 用 3 个真实场景,跟踪数据包从起点到终点的完整路径。
> 每一步标出**经过哪台设备、做什么处理**。

---

## 场景 1:行政办公室小王打开百度

**小王的电脑**:
- 网段:VLAN 200(`192.168.200.x/24`)
- 网关:192.168.200.1(N-core 的 SVI)
- 物理位置:某楼层,接在 N-HJ3 或 N-HJ4 接入交换机上

### 流量路径(7 跳)

```
第1跳  小王 PC(192.168.200.50)
        │
        │  ARP 请求:谁是 192.168.200.1 ?
        ▼
第2跳  接入交换机 N-HJ3 (U25/U26)
        │  Trunk 口透传到核心,不打 VLAN tag
        ▼
第3跳  锐捷 N-core(堆叠,U28+U29)
        │  • 看到目的 IP 是公网(8.8.8.8)
        │  • 查路由表:0.0.0.0/0 → 192.168.254.254
        ▼
第4跳  内网网关 EG3220 (U30)
        │  • 192.168.254.254 是它的 AggregatePort 1
        │  • 看到内网 IP → 触发 NAT(把 192.168.200.50 翻译成公网 IP)
        │  • 默认路由 → Gi0/5(172.31.254.1)→ 下一跳 172.31.254.2
        ▼
第5跳  核心防火墙 F1000-AK155 (U20)
        │  • G1/0/15 收到,识别为 Trust 区域
        │  • 查策略:Trust → Untrust 允许
        │  • 走 G1/0/4 → 192.168.5.1
        ▼
第6跳  出口路由器 MSR3620(不在本机柜,192.168.5.1)
        │  • NAT 出口
        │  • 走运营商 BGP / 静态路由
        ▼
第7跳  互联网 → 百度
```

**返回路径**:完全反向(百度 → MSR3620 → F1000 → EG3220 → N-core → HJ3 → PC)

**关键点**:
- 整条路过了 **6 台设备**
- EG3220 和 F1000 各自做 NAT(EG3220 做的是内网→内网,NAT 发生在 F1000 出口)
- F1000 防火墙在这里**不只是安全,也承担路由**

## 场景 2:医生在护士站调阅病人 CT 影像(PACS)

**医生电脑**:
- 网段:VLAN 30(`192.168.30.x/24`,所有科室)
- 网关:192.168.30.254(VRRP 虚拟 IP,主在 SW2)
- 物理位置:科室工位,接在 H3C 的本地 access 口

### 流量路径(5 跳)

```
第1跳  医生 PC(192.168.30.20)
        │
        │  ARP 请求:谁是 192.168.30.254 ?
        ▼
第2跳  H3C 核心 SW2(主,VLAN 30 优先级 110)
        │  • 收到 ARP,响应 SW2 的 MAC
        │  • 医生请求 https://pacs-server:443
        │  • 目的 IP 192.168.40.x(服务器区,VLAN 40)
        │  • 查路由表:192.168.40.0/24 → 192.168.3.2(F1000 G1/0/1)
        ▼
第3跳  核心防火墙 F1000-AK155 (U20)
        │  • G1/0/1 收到,VLAN 3 段
        │  • 识别源:Trust(192.168.30.0/24)
        │  • 识别目的:DMZ(VLAN 40)
        │  • 查策略:Trust → DMZ 允许(对象组 Client → Server)
        │  • 走 G1/0/2.40(子接口,192.168.40.254)
        ▼
第4跳  汇聚交换机 S5120V2 (U40)
        │  • 简单透传 VLAN 40
        ▼
第5跳  PACS 服务器(192.168.40.x)
        │
        │  返回影像数据
        ▼
   反向回流(SW2 → 医生 PC)
```

**关键点**:
- **只过了 3 台设备**(PC → SW2 → F1000 → 服务器),比场景 1 短
- **完全不走 EG3220,不经过 N-core,不经过 MSR3620**
- 防火墙在这里是**业务必经之路**(任何内网到服务器的访问都要过它)

## 场景 3:患者家属连 WiFi 上网

**访客手机**:
- 连上 SSID `Zangyiyuan`
- 密码:`XXK123123`(WPA2-PSK)
- 拿到 IP:192.168.236.x(VLAN 236 客户端地址池)

### 流量路径(6 跳)

```
第1跳  手机
        │
        │  WiFi 关联 + WPA2 认证
        ▼
第2跳  AP(吸顶/壁挂,具体型号未知)
        │  • CAPWAP 隧道 → AC(192.168.200.3)
        │  • 集中转发模式(forward central)
        ▼
第3跳  无线 AC WS6008 (U35-36)
        │  • 终结 CAPWAP 隧道
        │  • 给客户端分 IP(DHCP from W-core)
        │  • 数据从 Gi0/8 trunk 上行
        ▼
第4跳  W-core 锐捷 S5760C (U33)
        │  • VLAN 236 接口
        │  • DHCP 池:sta_pool → 192.168.236.0/22
        │  • 默认路由 → 192.168.255.254(EG3230)
        ▼
第5跳  外网网关 EG3230 (U37)
        │  • NAT 出口
        │  • Gi0/7 → ISP(10.2.2.1)
        ▼
第6跳  互联网
```

**关键点**:
- **完全独立的路径**,不碰 H3C 核心、不碰 N-core、不碰 F1000
- 这就是"内外网物理隔离"的实现:访客流量**根本没有进入内网**
- 集中转发 = 流量必须经 AC 解封装再转发,核心(W-core)压力大

## 关键洞察

### 1. F1000 防火墙是"内网枢纽"

只要内网用户访问服务器、互联网,**必须经过 F1000**。这就是为什么防火墙策略是整个网络安全的核心 —— 改错一条,可能全院断网或全院被黑。

### 2. 访客 WiFi 和办公 WiFi 是"两条路"

- 访客:AP → AC → W-core → EG3230(外网)
- 办公:AP → 锐捷 N-core(具体路径看 AP 接哪里)

如果办公 AP 也接在 W-core,那办公流量会和访客流量混在 W-core 上 → 这是混乱点之一(见 03)。

### 3. EG3220 是个"绕路"的网关

内网用户上网,本可以直接走 H3C 核心 → F1000 → MSR3620。
但实际是 N-core → EG3220 → F1000 → MSR3620。
**多绕了一步** —— 推测原因:
- N-core 是后建的办公网,需要单独 NAT/审计
- EG3220 提供流控、应用识别(锐捷 EG 系列擅长这个)
- F1000 防火墙不擅长高性能 NAT
