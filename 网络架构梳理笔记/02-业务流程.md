# 02 · 业务流程(流量怎么走的)

> **核心背景:旧院区(VLAN 10-40)和新院区(VLAN 200-205)并存,业务路径要分两条线追踪。**

---

## 场景 1:旧院区医生查 PACS(尚未迁移)

**医生电脑**:
- 网段:VLAN 30(`192.168.30.x/24`,所有科室)
- 网关:`192.168.30.254`(VRRP 虚拟 IP,主在 SW2)
- 物理位置:**旧院区**科室,接在 H3C 旧核心的本地 access 口

### 流量路径(5 跳)

```
第1跳  医生 PC(192.168.30.20)
        │
        ▼
第2跳  H3C 核心 SW2(VLAN 30 主,优先级 110)
        │  • 查路由表:192.168.40.0/24 → 192.168.3.2(F1000 G1/0/1)
        ▼
第3跳  核心防火墙 F1000-AK155
        │  • Trust → DMZ 策略放行
        │  • 走 G1/0/2.40
        ▼
第4跳  汇聚交换机 S5120V2
        ▼
第5跳  PACS 服务器(192.168.40.x)
```

**特点**:**完全走旧院区路径**,不经 N-core、不经 EG3220。

## 场景 2:新院区行政办公上网(已迁移)

**新院区员工电脑**:
- 网段:VLAN 200(`192.168.200.x/24`)
- 网关:`192.168.200.1`(N-core SVI)
- 物理位置:**新院区**工位,接在 N-HJ3 或 N-HJ4 接入交换机

### 流量路径(7 跳)

```
第1跳  员工 PC(192.168.200.50)
        │
        ▼
第2跳  接入交换机 N-HJ3 / N-HJ4
        │
        ▼
第3跳  锐捷 N-core(VSU 堆叠)
        │  • 查路由表:0.0.0.0/0 → 192.168.254.254(EG3220)
        ▼
第4跳  内网网关 EG3220
        │  • NAT:把 192.168.200.50 翻译成公网 IP
        │  • 默认路由 → Gi0/5(172.31.254.1)→ 172.31.254.2
        ▼
第5跳  核心防火墙 F1000-AK155(G1/0/15,Trust)
        │  • Trust → Untrust 放行
        │  • 走 G1/0/4
        ▼
第6跳  出口路由器 MSR3620(192.168.5.1)
        │
        ▼
第7跳  互联网
```

**特点**:**完全走新院区路径**,不经 H3C 旧核心。

## 场景 3:旧院区行政办公上网(还没迁)

**旧院区员工电脑**:
- 网段:VLAN 10(`192.168.10.x/24`,行政办公)
- 网关:`192.168.10.254`(VRRP 虚拟 IP,主在 SW1)
- 物理位置:**旧院区**工位,接在 H3C 旧核心

### 流量路径(待确认!)

这里有个**关键问题**:H3C 的路由表显示默认路由是 → `192.168.2.2`(F1000 G1/0/0)

```
旧院区 PC(VLAN 10)
  → H3C SW1
  → F1000 G1/0/0(Trust)
  → F1000 G1/0/4 → MSR3620 → 互联网
```

**问题**:
- 旧院区上网**直接走 F1000 防火墙**,没经过 EG3220
- EG3220 是**新院区专用网关**,只服务 VLAN 200-205
- 也就是说:**旧院区和新院区走的是不同的 NAT 出口**(F1000 vs EG3220)

**为什么这样设计?**
- 旧院区已经在 H3C 上跑了几十年,改 EG3220 风险太大
- 新院区从 0 建设,直接用 EG3220 走 NAT 更可控
- 等旧院区全部迁完,所有流量都会走 EG3220 → F1000 → MSR3620

## 关键洞察(更新版)

### 1. **新旧院区走的是不同的 NAT 出口**

```
旧院区:PC → H3C → F1000 → MSR3620(防火墙兼任 NAT)
新院区:PC → N-core → EG3220 → F1000 → MSR3620(EG3220 做 NAT)
```

**这意味着**:
- 旧院区用户**不经过 EG3220**,审计/流控不到
- 新院区用户的上网行为在 EG3220 上有完整记录
- 迁移完成后,所有用户都会经过 EG3220

### 2. **服务器区还在用旧 VLAN 40**

PACS、HIS 这些服务器**没有动**,还是接在 S5120V2 上,网段还是 `192.168.40.0/24`。

**所以两边医生看 PACS 都得经过 F1000 防火墙的 Trust→DMZ 策略**:
- 旧院区医生:H3C → F1000 → 服务器
- 新院区医生(如果已切):N-core → 防火墙 G1/0/15 → F1000 → 服务器

### 3. **F1000 防火墙是"新旧院区共用枢纽"**

迁移期 F1000 同时连接:
- 旧院区 H3C(G1/0/0 + G1/0/1)
- 新院区 EG3220(G1/0/15)
- 服务器 S5120V2(G1/0/2)
- 出口 MSR3620(G1/0/4)

**这台防火墙不能动**,任何策略调整都会同时影响新旧两区。

### 4. **DHCP 抢答只在迁移期出现**

- VLAN 10/20:H3C 和 N-core 都在宣告自己是 DHCP 服务器
- VLAN 200-205:只有 N-core 宣告(无冲突)
- VLAN 40:由 S5120V2 接管(若已配)

**等旧院区终端全部迁完,VLAN 10/20 就会从 H3C 上消失,冲突自然消除。**

---

## 一句话总结

> **迁移期最关键的不是"看哪里乱",而是"分清哪台设备在服务哪类业务"。**
> **H3C 在为旧院区服务,N-core 在为新院区服务,共用区两边都服务 —— 别把它们的流量路径混在一起画。**
