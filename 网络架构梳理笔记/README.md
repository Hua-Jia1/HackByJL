# 9 号机柜 — 网络架构梳理笔记

> **⚠️ 关键背景:这是一台"迁移中"的机柜。**
>
> - **旧院区** = VLAN 10-40(H3C 核心承载,**正在退役**)
> - **新院区** = VLAN 200-205(锐捷 N-core 承载,**正在接管**)
> - **共用区** = 防火墙/服务器/无线/远程(两边都用)
>
> 之前我以为这是个"混乱的网络",实际是**两套网络在过渡期内并存**。大部分"乱"都是**迁移期的正常状态**。

## 一页纸速查(One-Pager · 迁移版)

### 三块视角

```
┌────────── 旧院区(退役中) ──────────┐  ┌──── 新院区(接管中) ────┐
│ H3C SW1/SW2(VRRP 双机)              │  │ 锐捷 N-core(VSU 堆叠 2 台) │
│ VLAN 10 行政办公 192.168.10.x        │  │ VLAN 200 办公主 192.168.200.x│
│ VLAN 20 一级医技 192.168.20.x        │  │ VLAN 201 打印机 192.168.201.x│
│ VLAN 30 所有科室 192.168.30.x        │  │ VLAN 202 视频会议 192.168.202.x│
│ VLAN 40 服务器 192.168.40.x(共用)    │  │ VLAN 203-205 办公 192.168.20x│
│ 网关:192.168.10/20/30.254(VRRP)     │  │ 网关:192.168.200-205.1      │
└────────┬────────────────────────────┘  └─────────┬──────────────────┘
         │                                        │
         │ EG3220(新院区专用)                     │
         └────────────────┬───────────────────────┘
                          ↓
                ┌──────────────────────┐
                │ F1000 核心防火墙      │  ← 共用(新旧两区都过)
                │  Trust/DMZ/Untrust  │
                └──────────┬───────────┘
                           ↓
                ┌──────────────────────┐
                │ S5120V2 服务器汇聚    │  ← 共用
                │ VLAN 40(暂未迁)       │  ← HIS / PACS / OA
                └──────────────────────┘

────────────── 共用区(与新旧无关) ──────────────
  W-core(无线/外网核心) + 华为 USG6000E(远程)
  EG3230(外网)  +  MSR3620(出口,在别处)
```

### 关键认知

| 认知 | 解释 |
|------|------|
| H3C = 旧院区 | 正在退役,**别动它** |
| N-core = 新院区 | 正在接管,**新终端都插这边** |
| F1000 = 共用大门 | 任何跨区访问都过它,**不能动策略** |
| EG3220 = 新院区专用网关 | 旧院区用户**不经过**它 |
| EG3230 = 无线/外网专用 | 与院区无关 |
| 华为 USG = 远程 | 独立,与院区无关 |

### 3 个常见业务一句话

| 业务 | 院区 | 路径 |
|------|------|------|
| 旧院区医生看 PACS | 旧 | PC → H3C → F1000 → 服务器 |
| 新院区员工上网 | 新 | PC → N-core → EG3220 → F1000 → MSR3620 |
| 访客连 WiFi | 共用 | 手机 → AP → AC → W-core → EG3230 |

### 9 个问题(分类速记)

| 类别 | 问题 | 处置 |
|------|------|------|
| **A:迁移期正常现象**(4 个) | VLAN 10-40 在两台核心、DHCP 抢答、4 个"核心"挤一起、N-core VSU | **不要解决,等迁移完自然消失** |
| **B:独立问题**(5 个) | W-core 2 份配置、华为无文档、MSR3620 位置、命名混乱、VLAN 4 用途 | 单独处理 |
| **C:迁移卡点**(3 个) | 旧院区终端没盘点、NAT 策略没整合、服务器迁移没排 | **重点突破** |

### 5 阶段迁移路径

| 阶段 | 内容 | 风险 | 时机 |
|------|------|------|------|
| 0 | 摸清家底(SSH 导出、终端盘点) | 无 | **立即** |
| 1 | 命名规范(新院区设备) | 无 | 立即 |
| 2 | 制定切换计划(每台终端什么时候切) | 低 | 1 个月内 |
| 3 | 分批切换(每周 1-2 个 VLAN) | 中 | 1-3 个月 |
| 4 | H3C 旧核心退役(所有业务切完后) | 高 | 3-6 个月 |
| 5 | 清理归档(删旧 VLAN、写报告) | 无 | 退役后 |

---

## 三个"千万别做"

1. **别在迁移期硬删 N-core 上的 VLAN 10/20/30/40** —— 旧终端可能正在用
2. **别关 H3C 的 VRRP** —— 旧院区还在生产
3. **别在白天动服务器 VLAN 40** —— 全院业务会断
